# 学伴图谱项目级设计方案

本文档以根目录的《学伴项目-设计说明书》为最高设计方针。后续产品设计、前端重构、版本拆分和验收都必须围绕该说明书确立的路线执行：学伴不是普通学习社区，也不是 CRUD 资料库，而是以知识图谱画布为核心的学习平台。

## 1. 设计基准

### 1.1 产品定位

项目名称统一为 `StudyMate Graph`，中文名为“学伴图谱”。

产品定位：

```text
以知识图谱画布为核心的学习平台。
用户从资料、PDF、笔记、帖子和 AI 输出中提炼知识节点，
再通过可视化图谱组织知识、生成卡片并进入间隔复习。
```

核心学习闭环：

```text
资料收集 -> PDF 阅读 -> 批注摘录 -> 笔记沉淀 -> 图谱组织 -> AI 辅助理解 -> 卡片复习 -> 社区分享/讨论
```

### 1.2 总体原则

- 每个模块都要达到对应赛道“可用产品”的深度，不能停留在展示页或简单 CRUD。
- 图谱画布是核心差异化能力，但资料、阅读、笔记、复习、AI 和社区都必须能独立形成可用体验。
- 所有模块必须通过资料、笔记、图谱、卡片、AI、社区形成闭环，不做功能孤岛。
- 第一阶段允许分步实现，但最终设计必须能自然扩展到 Project Graph 级别的画布能力。
- AI 生成内容必须进入用户确认流程，不能直接覆盖用户内容。
- 第一阶段坚持单人图谱、Canvas 2D、自研轻量图对象模型，不做多人实时协作、CRDT 和 WebGL。

## 2. 模块对标与最低实现深度

| 模块 | 对标产品 | 学伴图谱目标 | MVP 不可低于 |
| --- | --- | --- | --- |
| 笔记/知识库 | Notion、Obsidian、Logseq、RemNote | 块编辑器、双链、版本、资料/图谱/卡片引用 | 富文本/Markdown、文件夹、标签、版本、引用关系 |
| 资料/PDF | Zotero、Readwise Reader、MarginNote、LiquidText | 资料管理、PDF 深度阅读、高亮批注、引用来源 | 资料 CRUD、PDF 阅读、高亮、批注、书签、阅读进度 |
| 图谱画布 | Project Graph、Miro、tldraw、draw.io | 学习知识图谱画布和工程图谱画布 | 节点、连线、拖拽、缩放、框选、历史、自动保存 |
| 卡片复习 | Anki、Quizlet、RemNote、Mochi | 从笔记、PDF、图谱节点生成卡片并复习 | Deck/Card、SM-2、今日复习、来源跳转 |
| AI 学伴 | NotebookLM、Notion AI、ChatGPT Projects | 基于资料和图谱的解释、总结、扩展、制卡 | AI 草稿、引用来源、用户确认、任务记录 |
| 社区 | Discourse、Reddit、Medium、小红书 | 学习内容发布、资料分享、问答讨论、治理 | 发帖、评论、点赞、收藏、举报、审核 |
| 后台运营 | Retool、Metabase、Discourse Admin | 内容治理、用户治理、AI 任务、运营配置 | 审核、举报、用户管理、系统配置、审计日志 |

## 3. 技术设计路线

### 3.1 主技术栈

```text
后端：Go + Gin + GORM + MySQL
前台：React + Vite + TypeScript
后台：Vue 3 + TypeScript + Element Plus
图谱画布：React + Canvas 2D + Zustand/Jotai + 自研图对象模型
富文本：Tiptap 或 Plate.js
PDF：PDF.js
AI：OpenAI 兼容接口
搜索：Meilisearch 起步，后续 Elasticsearch 或双引擎
文件：本地存储起步，后续 MinIO/OSS/COS
缓存与队列：Redis 起步，后续 Asynq/RabbitMQ/Redis Stream
```

### 3.2 数据职责

| 数据层 | 职责 | 第一阶段要求 |
| --- | --- | --- |
| MySQL | 权威业务事实、权限、审核、统计、关联索引 | 用户、资料、社区、笔记索引、图谱索引、卡片、AI 任务 |
| MongoDB | 复杂 JSON 内容、块文档、图谱文档、批注、AI 上下文 | `note_documents`、`note_snapshots`、`graph_documents`、`pdf_annotations` |
| Redis | 缓存、限流、验证码、会话状态、异步队列 | 登录限流、热点缓存、搜索同步队列 |
| Meilisearch | 第一阶段全文搜索 | 帖子、资料、笔记、图谱、卡片索引 |
| Elasticsearch | 复杂搜索、聚合统计、日志分析 | 后期升级或并行 |
| 向量数据库 | RAG 语义检索 | 后期规划，不进入第一阶段 |

### 3.3 图谱引擎边界

图谱画布不能依赖展示型 mindmap 库，应自研轻量图谱引擎：

```text
GraphCanvas
├── Camera               # 平移、缩放、坐标转换
├── StageObject          # 画布对象基类
├── Entity               # 节点、分组、图片、公式等实体
├── Association          # 直线边、曲线边、多目标边等关系
├── Renderer             # Canvas 渲染
├── Controller           # 鼠标、触控、键盘交互
├── HistoryManager       # 撤销/重做
├── LayoutEngine         # 自动布局
├── ImportExportEngine   # 导入导出
└── GraphAIEngine        # AI 生成和扩展图谱
```

第一阶段只做核心骨架和稳定数据模型；高级工程图、插件、桌面端、复杂协作后置。

## 4. 用户端设计方案

### 4.1 全局信息架构

用户端必须像长期学习工作台，而不是营销首页。

```text
左侧主导航：工作台 / 资料库 / 阅读器 / 笔记 / 图谱 / 复习 / AI 学伴 / 社区 / 设置
顶部工具条：全局搜索 / 最近任务 / AI 草稿入口 / 通知 / 用户菜单
中部工作区：当前任务主界面
右侧上下文：来源、引用、版本、批注、AI 建议、图谱入口
```

### 4.2 页面定位

- 工作台：最近资料、继续阅读、最近笔记、待复习、AI 草稿、社区动态。
- 资料库：筛选、资料列表、详情、元数据、上传、收藏、评分、阅读入口。
- 阅读器：资料切换、PDF 阅读区、页码跳转、高亮、批注、书签、摘录池。
- 笔记：笔记列表、富文本编辑、标签/文件夹、版本历史、来源资料、图谱引用。
- 图谱：画布核心工作区、节点工具栏、属性面板、搜索定位、自动保存状态。
- 复习：今日队列、Deck 总览、卡片编辑、来源回看、复习统计。
- AI 学伴：基于资料/笔记/图谱的总结、解释、扩展、制卡草稿和引用来源。
- 社区：内容流、话题、资料分享、图谱分享、问答讨论。

## 5. 管理端设计方案

管理端是治理后台，不是临时审核页。它必须服务内容安全、资料治理、AI 用量、用户管理和运营配置。

核心模块：

- 概览仪表盘。
- 内容审核和举报处理。
- 资料管理。
- 社区管理。
- 用户管理。
- 图谱模板。
- AI 任务和用量。
- 系统配置。
- 审计日志。

后台和用户端共用同一主题语言，但后台信息密度更高、装饰更少、更强调筛选、状态、批量操作和审计上下文。

## 6. 视觉与交互系统

### 6.1 产品气质

- 安静、专业、适合长时间停留。
- 有工具感，但不过分冰冷。
- 信息密度适中，关键路径明确。
- 以工作台和画布为中心，不做大 Hero 营销式首页。

### 6.2 主题方向

- 背景：浅暖灰 / 米白。
- 主文字：深墨绿灰。
- 主色：松柏绿。
- 辅色：暖琥珀。
- 危险色：砖红。
- 容器：大圆角、轻玻璃感、柔和投影。
- 功能卡片：小圆角、低对比边框。
- 按钮：胶囊按钮和轻描边按钮为主。

### 6.3 状态和占位要求

未实现模块也必须有高质量占位，明确说明：

- 模块定位。
- 将接入的真实能力。
- 与当前学习闭环的关系。
- 下一阶段入口。

不能出现大块空白、临时按钮堆叠或与主视觉系统脱节的占位页。

## 7. 实施顺序

### 阶段 A：统一壳层和设计系统

- 建立用户端学习工作台壳层。
- 建立管理端治理后台壳层。
- 统一前后台颜色、圆角、字号、阴影、边框、空状态、加载态和错误态。
- 用高质量占位承接图谱、复习、AI、课程等后续模块。

### 阶段 B：接回真实学习链路

- 资料库真实交互回接。
- PDF 阅读器真实交互回接。
- 批注、书签、阅读进度回接。
- 笔记富文本、版本历史和来源关系回接。
- MongoDB 内容文档双写逐步稳定。

### 阶段 C：图谱画布成为产品中枢

- 实现 Canvas 2D 画布、相机、节点、连线、历史和自动保存。
- 接通“批注/笔记/资料 -> 图谱节点草稿”。
- 接通“图谱节点 -> 卡片草稿”。
- 接入 AI 图谱解释和扩展，但所有生成内容都必须先确认。

### 阶段 D：复习、AI、搜索和运营完善

- 完成 SM-2 复习队列。
- 完成 AI 学伴任务、配额、引用来源和草稿确认。
- 接入 Meilisearch 全站搜索。
- 完善后台审核、举报、AI 任务、系统配置和运营统计。

## 8. 当前执行重点

当前阶段围绕 `v0.4.0` 与 `v0.5.0` 衔接推进：

- 继续加厚资料库、阅读器和笔记真实链路。
- 稳定笔记 MongoDB 内容文档和版本快照。
- 统一前后台设计 token 和组件语义。
- 补齐 PDF 页内高亮可视化和摘录池。
- 为图谱 MVP 预留稳定的前端状态模型、后端数据模型和入口。
- 优先接通“批注 -> 笔记 -> 图谱节点草稿”的最小闭环。
